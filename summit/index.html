// Hash-Based Detection
Sample 1 — Hash Block
SAMPLE 01
🔍
Malware Sandbox Analysis & SHA256 Block Rule

The room opens with an email from SPHINX (sphinx@pentesting.thm) who will work with me to conduct threat simulation and detection engineering tests. They have sent through the first malware sample asking me to scan sample1.exe with the Malware Sandbox tool and to find a unique way to detect and block it. Upon clicking on the attachment I am taken to the malware sandbox tool asking me to submit sample1.exe for analysis. Clicking the submit button starts the scan on a sandboxed Windows 10 x64 v1803 system and generates a report once finished.

The report shows the file is tagged as Trojan.Metasploit.A and includes three hash values — MD5 (cbda8ae000aa9cbe7c8b982bae006c2a), SHA1 (83d2791ca93e58688598485aa62597c0ebbf7610), and SHA256 (9c550591a25c6228cb7d74d970d133d75c961ffed2ef7180144859cc09efca8c). The behaviour analysis also flags METASPLOIT was detected under Malicious and Connects to unusual port under Suspicious. As MD5 and SHA1 are considered insecure I will use the SHA256 hash value to create a block rule. So I navigate to the Manage Hashes page, select the SHA256 radio box and add the hash value. Upon submitting I get a confirmation saying nice work — I have prevented sample1.exe from being executed. Check the inbox for the next steps.

// Network-Based Detection
Sample 2 — Firewall Rule
SAMPLE 02
🔥
C2 IP Block via Egress Firewall Rule

Going back to the inbox there is a new email from SPHINX titled "Update: You Blocked Me!" containing the first flag and introducing sample2.exe. SPHINX explains that while file hashes are the highest confidence indicators available, they are also the most susceptible to change — altering a single bit of the file generates a completely new hash, breaking the detection rule. To prove the point, SPHINX simply recompiled the malware and executed it without issue.

Clicking on the sample2.exe attachment takes me back to the Malware Sandbox. The report still tags it as Trojan.Metasploit.A but with a new file size of 202.73 KB and entirely different hash values. The behaviour analysis flags METASPLOIT was detected and under Suspicious: Connects to unusual IP address and Connects to unusual port (PID: 1927). The Network Activity shows 1 HTTP GET request from sample2.exe to 154.35.10.113:4444 with the full URL http://154.35.10.113:4444/uvLk8YI32, hosted by Intrabuzz Hosting Limited. So let's see if we can block this outbound connection with a firewall rule.

Clicking on the Firewall Rule Manager lets me create a rule. I select Egress for type as we want to block an outbound connection, set Source IP to Any to apply this rule to all IPs in the company, enter the suspicious destination IP of 154.35.10.113, and set Action to Deny. Upon clicking Save Rule the active rules table confirms the new egress deny rule is in place.

// FLAG 1 EMAIL — SPHINX confirms the hash block worked and introduces sample2.exe // MALWARE SANDBOX — Submitting sample2.exe for analysis // SANDBOX RUNNING — Analysis in progress // SANDBOX REPORT — C2 GET request to 154.35.10.113:4444 visible in network activity // FIREWALL RULE MANAGER — Existing active rules before creating the block rule // CREATE RULE — Egress deny rule targeting 154.35.10.113 // RULE CONFIRMED — sample2.exe blocked from reaching its C2 server

C2 Domain Block via DNS Filter Rule

Going back to the inbox there is a new email from SPHINX titled "Stumped again... for now!" containing the second flag and introducing sample3.exe. SPHINX acknowledges the IP block worked but points out it isn't bulletproof — they have signed up for a cloud service provider giving access to many more public IPs, with plenty of backups to failover to if they get blocked.

Clicking on the attachment takes me back to the Malware Sandbox. The report again tags as Trojan.Metasploit.A. The behaviour analysis flags METASPLOIT was detected and Downloads executable files from the Internet under Malicious (including backdoor.exe PID: 2172). Network activity shows 2 HTTP requests and 2 DNS requests. The HTTP requests are GET requests to emudyn.bresonicz.info:1337 — one fetching /km2993a and one downloading /backdoor.exe. The Connections table shows emudyn.bresonicz.info attributed to Xplorita Cloud Services — confirming SPHINX's cloud provider. So going back to the menu I navigate to the DNS Filter.

Clicking Create a Rule I fill in: Rule Name Block Malware Sample3, Category Malware, Domain Name emudyn.bresonicz.info, and Action Deny. Clicking Save Rule adds the new rule to the active rules table and a confirmation message appears.

Sample 4 — Registry Modification
SAMPLE 04
📝
Sigma Rule — DisableRealtimeMonitoring Registry Write

Going back to the inbox there is a new email from SPHINX titled "RE: Stumped again... for now!" containing the third flag and introducing sample4.exe. SPHINX acknowledges the DNS block worked but explains they now have to purchase and register new domain names. This time SPHINX makes it clear — blocking hashes, IPs, or domains won't help. Instead we need to consider what artefacts (or changes) the malware leaves on the victim's host system.

The Sandbox report shows the file this time has no tags — SPHINX has stripped the obvious malware signatures. File size is 219.46 KB. The behaviour analysis flags Disables Windows Defender Real-time monitoring and Downloads executable files from the Internet under Malicious. Under Suspicious: Connects to unusual IP address, Connects to unusual port, and notably Makes changes to the registry. Scrolling to the bottom of the report reveals a Registry Activity section — 3 total events (1 read, 2 write). The key modification event shows sample4.exe (PID: 3806) performing a write to HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows Defender\Real-Time Protection with Name DisableRealtimeMonitoring and Value 1.

Since blocking hashes, IPs, or domains won't work I use the Sigma Rule Builder. At Step 1 I select Sysmon Event Logs. At Step 2 I select Registry Modifications. Step 3 asks for the rule conditions:

// Step 3: Registry Modifications Registry Key: HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows Defender\Real-Time Protection Registry Name: DisableRealtimeMonitoring Value: 1 ATT&CK ID: Defense Evasion (TA0005)

Clicking Validate Rule generates the Sigma rule titled "Modification of Windows Defender Real-Time Protection" (ID: windows_registry_defender_disable_realtime), using EventID 4663, mapped to attack.ta0005, rated high severity.

// Sigma Rule — Beaconing
Sample 5 — Network Connections
SAMPLE 05
📡
Sigma Rule — Beaconing Pattern Detection

Going back to the inbox there is a new email from SPHINX titled "New Approach" — signed off as "Annoyed Sphinx" — containing the fourth flag and introducing sample5.exe. SPHINX admits the registry detection threw a serious wrench into their operation. For this new sample all "heavy lifting" occurs on the back-end server, meaning protocols and host artefacts can be changed at will. To help, SPHINX attaches the outgoing_connections.log from the last 12 hours of network connections on the victim machine.

Opening the log in the Attachment Viewer reveals a clear beaconing pattern — one destination IP of 51.102.10.19 on port 443 appears repeatedly with a consistent packet size of exactly 97 bytes, occurring every 30 minutes. The other entries to different IPs and varying byte sizes are noise — normal traffic.

Going back to the Sigma Rule Builder and clicking Create Sigma Rule, I again select Sysmon Event Logs at Step 1. At Step 2 I select Network Connections. At Step 3, because SPHINX can rotate IPs and ports, I set both Remote IP and Remote Port to Any and rely on the consistent beacon signature instead:

// Step 3: Network Connections Remote IP: Any Remote Port: Any Size (bytes): 97 Frequency (seconds):1800 ATT&CK ID: Command and Control (TA0011)

Clicking Validate Rule generates the Sigma rule titled "Alert on Suspicious Beacon Network Connections" (ID: network_connections_criteria_sysmon), using EventID 3, wildcarded RemoteIP and RemotePort, mapped to attack.ta0011, rated high severity.

// FLAG 4 EMAIL — "New Approach" signed "Annoyed Sphinx" — confirms registry detection worked, attaches outgoing_connections.log // OUTGOING CONNECTIONS LOG — Beaconing to 51.102.10.19 at 97 bytes every 30 minutes clearly visible among normal traffic noise // SIGMA RULE BUILDER — Returning to build a new rule for sample5.exe // STEP 3 — Empty Network Connections form showing required fields // STEP 3 COMPLETED — Any/Any for IP and port, 97 bytes, 1800 second frequency, TA0011 // SIGMA VALIDATION — "Alert on Suspicious Beacon Network Connections", EventID 3, high severity, TA0011

// Sigma Rule — TTP Detection
Sample 6 — File Creation / Exfiltration
SAMPLE 06
📂
Sigma Rule — Exfiltration Staging File Detection

Going back to the inbox there is a new email from SPHINX titled "RE: New Approach" — now signed as "Very Annoyed Sphinx! 😤" — containing the fifth flag and introducing sample6.exe. SPHINX admits the beaconing detection was impressive. They can keep this up one or two more times, but the reward no longer outweighs the cost and they would rather find an easier target lower on the pyramid. For this final sample, artefacts and tool detection won't be enough — we need to focus on something extremely hard to change subconsciously: their techniques and procedures. SPHINX attaches the commands.log recorded from all previous samples.

Opening the commands.log shows every command SPHINX runs on compromised machines — all output is piped to the same staging file %temp%\exfiltr8.log:

dir c:\ >> %temp%\exfiltr8.log dir "c:\Documents and Settings" >> %temp%\exfiltr8.log dir "c:\Program Files\" >> %temp%\exfiltr8.log dir d:\ >> %temp%\exfiltr8.log net localgroup administrator >> %temp%\exfiltr8.log ver >> %temp%\exfiltr8.log systeminfo >> %temp%\exfiltr8.log ipconfig /all >> %temp%\exfiltr8.log netstat -ano >> %temp%\exfiltr8.log net start >> %temp%\exfiltr8.log

The pattern is clear — SPHINX enumerates directory listings, local admin accounts, OS version, system info, network config, active connections, and running services, all piped into exfiltr8.log in the temp folder. This is a consistent TTP that won't change regardless of IP, domain, hash, or protocol used. So going back to the Sigma Rule Builder, I select Sysmon Event Logs at Step 1, then File Creation and Modification at Step 2. Step 3 conditions:

// Step 3: File Creation and Modification File Path: %temp% File Name: exfiltr8.log ATT&CK ID: Exfiltration (TA0010)

Clicking Validate Rule generates the final Sigma rule titled "Alert on Potential Exfiltration through File Creation or Modification" (ID: sysmon_file_creation_modification_temp), using EventID 2, tagged with attack.ta0010, attack.exfiltration, attack.file_creation, and attack.file_modification, rated high severity. The rule is automatically deployed to PicoSecure's SIEM. Room complete!

// FLAG 5 EMAIL — "RE: New Approach" signed "Very Annoyed Sphinx!" — confirms beaconing detection, attaches commands.log // COMMANDS.LOG — All 10 enumeration commands consistently output to %temp%\exfiltr8.log // SIGMA RULE BUILDER — Returning for the final rule of the room // STEP 2 — Selecting File Creation and Modification as the target Sysmon event type // STEP 3 — Empty File Creation form // STEP 3 COMPLETED — %temp%, exfiltr8.log, Exfiltration (TA0010) // SIGMA VALIDATION — "Alert on Potential Exfiltration through File Creation or Modification", EventID 2, four ATT&CK tags, high severity. Room complete!
