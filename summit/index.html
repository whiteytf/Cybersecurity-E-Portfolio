// Room Scenario

After participating in one too many incident response activities, PicoSecure has decided to conduct a threat simulation and detection engineering engagement to bolster its malware detection capabilities. You have been assigned to work with an external penetration tester in an iterative purple-team scenario. The tester will be attempting to execute malware samples on a simulated internal user workstation. At the same time, you will need to configure PicoSecure's security tools to detect and prevent the malware from executing.

Following the Pyramid of Pain's ascending priority of indicators, your objective is to increase the simulated adversaries' cost of operations and chase them away for good. Each level of the pyramid allows you to detect and prevent various indicators of attack.

SAMPLE 01 Hash-Based Blocking — sample1.exe File Hash

The room opens with an email from SPHINX ([email protected]) who will work with me to conduct threat simulation and detection engineering tests. They have sent through the first malware sample asking me to scan sample1.exe with the Malware Sandbox tool and to find a unique way to detect and block it. Upon clicking on the attachment I am taken to the malware sandbox tool asking me to submit sample1.exe for analysis. Clicking the submit button starts the scan on a sandboxed Windows 10 x64 v1803 system and generates a report once finished.

The report shows the file is tagged as Trojan.Metasploit.A and includes three hash values — MD5 (cbda8ae000aa9cbe7c8b982bae006c2a), SHA1 (83d2791ca93e58688598485aa62597c0ebbf7610), and SHA256 (9c550591a25c6228cb7d74d970d133d75c961ffed2ef7180144859cc09efca8c). The behaviour analysis also flags METASPLOIT was detected under Malicious and Connects to unusual port under Suspicious. As MD5 and SHA1 are considered insecure I will use the SHA256 hash value to create a block rule. So I navigate to the Manage Hashes page, select the SHA256 radio box and add the hash value. Upon submitting I get a confirmation saying nice work — I have prevented sample1.exe from being executed. Check the inbox for the next steps.

SAMPLE 02 Firewall Egress Rule — sample2.exe Network Block

Going back to the inbox there is a new email from SPHINX titled "Update: You Blocked Me!" containing the first flag and introducing sample2.exe. SPHINX explains that while file hashes are the highest confidence indicators available, they are also the most susceptible to change — altering a single bit of the file generates a completely new hash, breaking the detection rule. To prove the point, SPHINX simply recompiled the malware and executed it without issue.

Clicking on the sample2.exe attachment takes me back to the Malware Sandbox. Submitting the sample starts the scan on a Windows 10x64 v1803 system and generates a report. The report shows the file is still tagged as Trojan.Metasploit.A with the following details:

  • File Size: 202.73 KB
  • File Type: PEX - PE32+ executable (GUI) x86-64, for MS Windows
  • MD5: 4d661bf6d9dfdb8bf5915d513b572a6bd
  • SHA1: 687897b974c27c8547cfc5acc90fb28ad2f0e975
  • SHA256: d576245e05e6b752b2fdffe43abeab1b2e138355bb0169fd04924d6cebc1cdf9

As we cannot rely on hashes we need to find another way to detect and block. Looking at the Behaviour Analysis section the Suspicious column flags both Connects to unusual IP address and Connects to unusual port for sample2.exe (PID: 1927). Looking at the Network Activity section we can see 1 HTTP request, 3 TCP/UDP connections and a GET request from PID 1927 to 154.35.10.113:4444 with the full URL http://154.35.10.113:4444/uvLk8YI32, hosted by Intrabuzz Hosting Limited. So let's see if we can block this outbound connection with a firewall rule.

Clicking on the Firewall Rule Manager lets me create a rule. I select Egress for type as we want to block an outbound connection, set Source IP to Any to apply this rule to all IPs in the company, enter the suspicious destination IP of 154.35.10.113, and set Action to Deny. Upon clicking Save Rule the active rules table confirms the new egress deny rule is in place and a confirmation message appears — the firewall rule prevented sample2.exe from connecting to the command-and-control server.

Submitting sample2.exe to the Malware Sandbox for analysis.

Completed sandbox report — Trojan.Metasploit.A detected, with network activity showing the GET request to 154.35.10.113:4444.

The Firewall Rule Manager showing the existing active rules before creating the new block rule.

Creating the egress deny rule targeting 154.35.10.113.

Rule saved and confirmed — sample2.exe blocked from reaching its C2 server.

SAMPLE 04 Sigma Rule — Registry Modification (sample4.exe) TA0005 — Defence Evasion

Going back to the inbox there is a new email from SPHINX titled "RE: Stumped again... for now!" containing the third flag and introducing sample4.exe. SPHINX acknowledges the DNS block worked but explains they now have to purchase and register new domain names and modify DNS records. This time SPHINX makes it clear — blocking hashes, IPs, or domains won't help. Instead we need to consider what artefacts (or changes) the malware leaves on the victim's host system.

Clicking on the attachment takes me back to the Malware Sandbox. Submitting the sample starts the scan on a Windows 10x64 v1803 system. The report shows the file this time has no tags — SPHINX has stripped the obvious malware signatures. File details from the report:

  • File Size: 219.46 KB
  • File Type: PEXE-PTES - Runnable (CLI) x86-64, for MS Windows
  • Tags: None

The Behaviour Analysis flags Disables Windows Defender Real-time monitoring and Downloads executable files from the Internet under Malicious (sample4.exe PID: 3806 and backdoor.exe PID: 1367). Under Suspicious we see Connects to unusual IP address, Connects to unusual port, and notably Makes changes to the registry (sample4.exe PID: 3806). Network activity shows 2 HTTP requests, 3 TCP/UDP connections, and 1 DNS request to the domain cranedr8.liveworks.syt resolving to 160.23.25.118, all via Xplorita Cloud Services.

Scrolling to the bottom of the report reveals a Registry Activity section — 3 total events (1 read, 2 write, 0 delete). The modification events show the most important artefact: sample4.exe (PID: 3806) performing a write operation to the key HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows Defender\Real-Time Protection with Name DisableRealtimeMonitoring and Value 1 — disabling Windows Defender. Since blocking hashes, IPs, or domains won't work we need to use the Sigma Rule Builder.

Clicking on the Sigma Rule Builder shows it is powered by Sigma GPT and will automatically deploy validated rules to PicoSecure's SIEM solution. Clicking Create Sigma Rule presents Step 1 — I select Sysmon Event Logs as we know sample4.exe is modifying the registry on the victim's machine and Sysmon covers process creations, network connections, file creation, and more. Step 2 presents four Sysmon event options — I select Registry Modifications as this detects changes to registry keys or values such as system settings and security policies. Step 3 then asks for the rule conditions:

  • Registry Key: HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows Defender\Real-Time Protection
  • Registry Name: DisableRealtimeMonitoring
  • Value: 1
  • ATT&CK ID: Defense Evasion (TA0005)

Clicking Validate Rule generates the Sigma rule and displays it in the Sigma Rule Validation panel. The generated rule is titled "Modification of Windows Defender Real-Time Protection" with ID windows_registry_defender_disable_realtime, using EventID 4663, ObjectType Key, and a level of high. The rule also notes legitimate changes to Windows Defender settings as a false positive to be aware of.

SAMPLE 05 Sigma Rule — Beaconing Behaviour (sample5.exe) TA0011 — C2

Going back to the inbox there is a new email from SPHINX titled "New Approach" containing the fourth flag and introducing sample5.exe. SPHINX signs off as "Annoyed Sphinx" — admitting the registry detection threw a serious wrench into their operation and that developing new techniques was a significant cost and time investment. However, SPHINX explains that for this new sample all of the "heavy lifting" and instruction occurs on the back-end server, meaning they can easily change the types of protocols used and the artefacts left on the host. To help, SPHINX attaches the outgoing_connections.log from the last 12 hours of network connections on the victim machine.

Opening the attachment in the Attachment Viewer shows the log file. Looking at the connection log we can see a clear pattern — one destination IP of 51.102.10.19 on port 443 appears repeatedly with a consistent packet size of exactly 97 bytes. Looking at the timestamps we can see this connection occurs reliably every 30 minutes. The other entries to different IPs and varying byte sizes are noise — normal traffic. This regular, same-size, same-destination traffic is classic beaconing behaviour.

Going back to the Sigma Rule Builder and clicking Create Sigma Rule, I again select Sysmon Event Logs at Step 1. At Step 2 I select Network Connections — this detects outgoing network connections, traffic patterns, and connections made by specific processes. Step 3 asks for the rule conditions. The rule will detect connections based on remote IP, port, size, and frequency. Because SPHINX told us they can change protocols and IPs, I set both Remote IP and Remote Port to Any so the rule isn't tied to specific values that can be rotated, and rely on the consistent beacon signature instead:

  • Remote IP: Any
  • Remote Port: Any
  • Size (bytes): 97
  • Frequency (seconds): 1800
  • ATT&CK ID: Command and Control (TA0011)

Clicking Validate Rule generates the Sigma rule. The validated rule is titled "Alert on Suspicious Beacon Network Connections" with ID network_connections_criteria_sysmon, using EventID 3, RemoteIP and RemotePort both set to wildcard *, Size 97, and Frequency 1800 seconds. It maps to attack.ta0011 and is rated high severity. The false positive note flags that legitimate network traffic may match this criteria.

Flag 4 email from SPHINX — titled "New Approach", signed as "Annoyed Sphinx", confirming the registry detection worked and introducing sample5.exe with the outgoing_connections.log attachment.

The outgoing_connections.log viewed in the Attachment Viewer — the beaconing pattern to 51.102.10.19 at 97 bytes every 30 minutes is clearly visible among the noise.

The Sigma Rule Builder landing page — returning to build a new rule for sample5.exe.

Step 3 — empty Network Connections form showing the required fields: Remote IP, Remote Port, Size, Frequency and ATT&CK ID.

Step 3 completed — Network Connections rule configured with Any/Any for IP and port, 97 bytes, 1800 second frequency, and TA0011 Command and Control.

Sigma Rule Validation — rule titled "Alert on Suspicious Beacon Network Connections", EventID 3, wildcarded IP and port, rated high severity and mapped to TA0011.

SAMPLE 06 Sigma Rule — File Creation / Exfiltration (sample6.exe) TA0010 — Exfiltration

Going back to the inbox there is a new email from SPHINX titled "RE: New Approach" containing the fifth flag and introducing sample6.exe. SPHINX — now signing off as "Very Annoyed Sphinx! 😤" — admits the beaconing detection was impressive and that going back to the drawing board to build a brand new tool would be another significant cost and time investment. SPHINX states they can keep this up one or two more times, but the reward no longer outweighs the cost and they would rather find an easier target lower on the pyramid. For this final sample SPHINX warns that artefacts and tool detection won't be enough — we need to focus on something extremely hard to change subconsciously: their techniques and procedures. To help, SPHINX attaches the commands.log recorded from all previous samples showing the exact actions they tend to perform on victims once remote access is established.

Opening the commands.log in the Attachment Viewer shows the full list of commands SPHINX runs on compromised machines. Every command outputs information from the victim machine to the same staging file — %temp%\exfiltr8.log:

  • dir c:\ >> %temp%\exfiltr8.log
  • dir "c:\Documents and Settings" >> %temp%\exfiltr8.log
  • dir "c:\Program Files\" >> %temp%\exfiltr8.log
  • dir d:\ >> %temp%\exfiltr8.log
  • net localgroup administrator >> %temp%\exfiltr8.log
  • ver >> %temp%\exfiltr8.log
  • systeminfo >> %temp%\exfiltr8.log
  • ipconfig /all >> %temp%\exfiltr8.log
  • netstat -ano >> %temp%\exfiltr8.log
  • net start >> %temp%\exfiltr8.log

The pattern is clear — SPHINX enumerates directory listings, local administrator accounts, OS version, system information, network configuration, active connections, and running services — all piped into exfiltr8.log in the temp folder before exfiltration. This is a consistent TTP that won't change regardless of what IP, domain, hash, or protocol is used. So going back to the Sigma Rule Builder and clicking Create Sigma Rule, I again select Sysmon Event Logs at Step 1. At Step 2 I select File Creation and Modification — this detects files being created or modified, changes to critical system files, or creation of executables or scripts. Step 3 asks for the rule conditions:

  • File Path: %temp%
  • File Name: exfiltr8.log
  • ATT&CK ID: Exfiltration (TA0010)

Clicking Validate Rule generates the final Sigma rule and triggers a congratulations notification. The validated rule is titled "Alert on Potential Exfiltration through File Creation or Modification" with ID sysmon_file_creation_modification_temp. It uses EventID 2, targets TargetFilename *\\exfiltr8.log and TargetPath *\\AppData\\Local\\Temp\\*, and carries tags for attack.ta0010, attack.exfiltration, attack.file_creation, and attack.file_modification. The rule is rated high severity with a false positive note for legitimate use of file creation in a user's temp folder. The rule is automatically deployed to PicoSecure's SIEM system, completing the room.

Flag 5 email from SPHINX — titled "RE: New Approach", signed as "Very Annoyed Sphinx!", confirming the beaconing detection worked, warning that TTPs are the final challenge, and attaching commands.log.

The commands.log viewed in the Attachment Viewer — all 10 enumeration commands consistently output to %temp%\exfiltr8.log, revealing SPHINX's post-exploitation TTP.

The Sigma Rule Builder landing page — returning for the final rule of the room.

Step 2 — selecting File Creation and Modification as the target Sysmon event type.

Step 3 completed — file path %temp%, file name exfiltr8.log, and ATT&CK ID set to Exfiltration (TA0010).

Sigma Rule Validation — rule titled "Alert on Potential Exfiltration through File Creation or Modification", EventID 2, four ATT&CK tags, rated high severity. Room completed!
