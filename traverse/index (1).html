ID Parameter Enumeration

By incrementing the id parameter starting at id=1, each request returned a different customer record. There was no access control validating which users could view which records — a classic IDOR vulnerability.

http://<target-ip>/planning/api?id=1 ← start here http://<target-ip>/planning/api?id=2 http://<target-ip>/planning/api?id=3 ← admin account + /realadmin dir http://<target-ip>/planning/api?id=4 http://<target-ip>/planning/api?id=5 ← target email found

Key findings from enumeration:

• id=3 — Identified as the admin account, leaked a /realadmin directory path along with credentials.

• id=5 — Revealed the target email: [email protected]

// Reflection
Key Takeaways
🧠
Vulnerabilities Identified

Sensitive data in client-side JS — Secrets should never be embedded in JavaScript files served to the browser, even if obfuscated or encoded.

Directory listing enabled — Web servers should be configured to deny directory browsing to prevent file enumeration.

Insecure Direct Object Reference (IDOR) — API endpoints must validate that the authenticated user is authorised to access the requested resource, not just that they are logged in.

Client-side only access controls — Command execution gated only by a UI dropdown with no server-side validation is trivially bypassed via DOM manipulation.

OWASP A01 — Broken Access Control OWASP A02 — Cryptographic Failure OWASP A05 — Security Misconfiguration
