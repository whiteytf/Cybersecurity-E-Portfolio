Source Analysis & Flag 1
STEP 01
🔍
VIEW SOURCE → JavaScript Enumeration

Starting with a basic View Page Source on the target webpage revealed two JavaScript files being loaded — tailwind.min.js and the suspicious custom.min.js. The commented-out /logs directory reference was also visible, left in by the DevOps team.

Opening custom.min.js directly in the browser revealed what appeared to be an encoded string. The encoding pattern was consistent with hexadecimal encoding.

View Source JS Enumeration Hex Encoding
STEP 02
🧪
CyberChef Decoding

The encoded string from custom.min.js was pasted into CyberChef and the From Hex recipe was applied, successfully decoding the value to reveal the first flag.

// HEX ENCODED CONTENT IN custom.min.js

The file contained a comment left by the attacker — // I WILL KEEP THE OBFUSCATED SO NO ONE CAN UNDERSTAND — followed by a large block of space-separated hex values. This confirmed hex encoding was being used to hide the malicious code from developers inspecting the file.

// CYBERCHEF FROM HEX DECODE

Decoding the hex in CyberChef using the From Hex recipe with a Space delimiter revealed a deobfuscated JavaScript function. The output contained several variables that together formed the flag:

var n = "DIRECTORY"; var e = "LISTING"; var os = "IS THE"; var i = "ONLY WAY"; ↳ console.log("Flag:" + n + " " + e + " " + os + " " + i) ↳ Flag: DIRECTORY LISTING IS THE ONLY WAY
// ROOM QUESTIONS
Q: What type of encoding is used by the hackers to obfuscate the JavaScript file?
A: HEX
Q: What is the flag value after deobfuscating the file?
A: THM{directory_listing_is_the_way}

Directory Listing & Credential Discovery
STEP 03
📂
Exposed /logs Directory

Returning to the page source revealed a reference to a /logs directory. Navigating directly to it confirmed directory listing was enabled — a misconfiguration that exposed server files to anyone who knew the path.

http://10.48.130.32/logs/ ↳ Apache/2.4.41 (Ubuntu) — directory listing enabled ↳ email_dump.txt — 2023-05-26 06:11 — 450 bytes

// INDEX OF /logs — DIRECTORY LISTING EXPOSED

The Apache server had directory listing enabled, meaning anyone who knew or guessed the path could browse its contents freely. The exposed file email_dump.txt was only 450 bytes — small enough to be an internal message — and dated 2023-05-26, suggesting it was left behind during the QA-to-Production migration. The server footer also revealed the underlying technology stack: Apache/2.4.41 running on Ubuntu at 10.48.130.32 Port 80 — useful intelligence for further exploitation.

// IDOR
Insecure Direct Object Reference
STEP 05
🔑
Accessing the /planning Directory

Navigating to the /planning directory and submitting the key THM{100100111} granted access to the Tourism MHT API documentation page. This page openly documented how to query customer records — with no authentication beyond the directory password.

http://10.48.130.32/planning/ Password: THM{100100111}

// /planning — API DOCUMENTATION PAGE

The page revealed the full API specification:

• Base URL: MACHINE IP Address

• Endpoint: api/?customer_id=1

• HTTP Method: GET

• Parameter: {id} — the customer ID for retrieving customer information

The example API response schema exposed exactly what each record would return — critically including sensitive fields with no access control:

{ "data": { "id": "1", "name": "string", "email": "example@example.com", "password": "******", ← stored in plaintext, exposed per-record "timestamp": "2023-05-23 04:47:25", ← real date seen across records "role": "user" or "admin", ← role fully disclosed "loginURL": "/client" or "/realadmin", ← leaks admin panel path "isadmin": "0" or "1" ← trivially identifies admin accounts }, "response_code": 200, "response_desc": "Success" }

The password, role, loginURL and isadmin fields were particularly dangerous — passwords were stored and returned in plaintext, and by simply incrementing customer_id it was possible to identify any admin account, retrieve their password, and get the exact login path — all in a single unauthenticated GET request.

Auth Bypass API Discovery Sensitive Data Exposure
STEP 06
🔢
ID Parameter Enumeration

By incrementing the customer_id parameter starting at customer_id=1, each request returned a different customer record. There was no access control validating which users could view which records — a classic IDOR vulnerability. The response schema had already revealed that each record would expose role, loginURL and isadmin fields, making admin enumeration trivial.

http://10.48.130.32/api/?customer_id=1 ← start here http://10.48.130.32/api/?customer_id=2 http://10.48.130.32/api/?customer_id=3 ← isadmin: true + loginURL → /realadmin http://10.48.130.32/api/?customer_id=4 http://10.48.130.32/api/?customer_id=5 ← john@traverse.com

Key findings from enumeration:

// API RESPONSE — customer_id=5

• customer_id=5 — John, email john@traverse.com, password qwerty5, role user, loginURL /client, isadmin 0. Timestamp 2023-05-23 04:47:25.

// API RESPONSE — customer_id=3 (ADMIN)

• customer_id=3 — admin, email realadmin@traverse.com, password admin_key!!!, role admin, loginURL /realadmin, isadmin 1. The plaintext password and admin login path were both returned directly in the API response with no authorisation check whatsoever.

── ID 5 (John) ────────────────────────── email john@traverse.com password qwerty5 role user | isadmin: 0 loginURL /client ── ID 3 (Admin) ───────────────────────── email realadmin@traverse.com password admin_key!!! ← plaintext! role admin | isadmin: 1 loginURL /realadmin ← admin panel path

STEP 07

Admin Panel — Command Execution

Navigating to /realadmin presented a login page. Using the credentials leaked from the API at id=3 granted access to an admin management panel.

The panel exposed a command execution dropdown containing two options. Opening DevTools → Elements tab revealed the underlying option values:

<option value="whoami">whoami</option> <option value="pwd">pwd</option>

By right-clicking the dropdown and selecting Edit as HTML in DevTools, a third option was injected:

<option value="ls -la">ls -la</option> ↳ Injected directly into the DOM — no server-side validation

Key Takeaways
🧠
Vulnerabilities Identified

Sensitive data in client-side JS — Secrets should never be embedded in JavaScript files served to the browser, even if obfuscated or encoded.

Directory listing enabled — Web servers should be configured to deny directory browsing to prevent file enumeration.
